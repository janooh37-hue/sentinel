1
398 of 724 active leave records are duplicates (55%). 🧮Data integrity & correctness

Bugs the browser surfaced that pure code review would miss — the code paths look correct, but the live data proves otherwise.

D1 Duplicate leave records — 398 of 724 active rows (dedup guard too narrow)

HighDatabaseLivecorrectness
WhatGenerating a leave form inserts a Leave row. Re-generating the same leave (double-submit, separate session, retry) inserts another. Employee G4481 has two identical "Sick Leave · Jul 1–3 · Approved" rows (IDs 751 & 752) — both visible on the Dashboard's "On leave today" and "Returning in 7 days" lists, and the employee page shows "6d taken" for a single 3-day leave.
Evidence65 duplicate (employee, type, start, end) groups; 398 excess rows. The 2026-03-26 incident left employee G4604 with 300 identical rows. There is a WF-03 dedup guard, but it only matches rows created >= now-2min; IDs 751/752 were created 5.7 min apart and slipped through. 29 duplicate groups span >2 min — structurally invisible to the guard.
ImpactLeave balances double-count; the Leaves annual report (1210 sick days, 2900 total) and every Dashboard figure are inflated; approvals show phantom entries.
Plan
  • Widen the guard: dedup on (employee, type, overlapping date range) regardless of creation time — a truly new leave differs in dates.
  • Add a DB uniqueness backstop: partial unique index on (employee_id, leave_type, start_date, end_date) WHERE deleted_at IS NULL.
  • One-time cleanup migration to soft-delete the 398 duplicates (keep lowest id per group), re-pointing any Document.leave_id references first.
document_service.py:1469-1512 (guard) · leave_service.py:250-283 · verified in data/gssg.db

D2 Document Expiry page & "Expiring Soon" widget are functionally dead

HighDatabaseLivedead feature
WhatThe /expiry page and the Dashboard widget filter on Emirates-ID / passport expiry dates that are never populated.
Evidenceuae_id_expiry, passport_expiry, passport_no are NULL/empty for all 300 employees (only uae_id_no is filled, 299/300). Live: the page shows "No expiring documents match your filters" and "0" for every window; the widget always renders empty.
ImpactA whole navigable feature and a dashboard slot that can never do anything — reads as "nothing expiring" when really "no data". A misplaced sense of safety for a compliance-critical field.
Plan
  • Decide the data source: have the OCR/intake pipeline extract ID & passport expiry into these columns (they're already extracted onto scans), or add them to the employee edit form + roster import.
  • Until a source exists, hide the Expiry route and widget behind a "has any expiry data" guard so the app doesn't advertise a dead surface.
  • Also fix the over-fetch (see P3) once data lands.
expiry.py:20 · ExpiringSoonWidget.tsx:85-91 · columns in employees table

D3 Leave status & type stored in two incompatible formats

HighDatabaseCodeold data / correctness
WhatOlder rows store bilingual concatenations — leave_type = "Sick Leave - الإجازة المرضية", status = "Generated - تم الإنشاء" — while newer code writes canonical "Sick Leave" / "Approved". Both coexist.
Evidence623/724 leave_type and 714/724 status values carry the " - " bilingual suffix. Distinct statuses include both "Approved" and "Approved - موافق". Visible on the employee Leaves tab (mixed formats in one list).
ImpactEvery consumer must funnel through canonStatus(); where one forgets (the employee LeavesTab, see C3) the status pill falls back to a neutral colour and text renders untranslated. Analytics grouping is fragile.
Plan
  • Normalize at rest: migration to rewrite leave_type/status to canonical codes; render the bilingual label at the UI layer from a lookup, not from stored text.
  • Enforce canonical values on write (enum/validation in leave_lifecycle).
leave_lifecycle.py (canonical_status) · LeavesTab.tsx:15-20 · leaves table

D4 370 of 378 Book-register records are stranded drafts

MediumDatabaseLivedata hygiene
WhatThe Records register shows 378 entries, 370 of them in the "Drafts" bucket (approval_state = 'none'). Only 6 approved, 1 returned, 1 awaiting-copy.
ImpactThe register — meant as a ref-number log of real correspondence — is 98% noise. The "Drafts (370)" segment and the Dashboard book_draft_count make the backlog look like pending work when it's mostly abandoned auto-created drafts.
Plan
  • Clarify lifecycle: is a document-generated book meant to stay a "draft" forever? If drafts are transient, auto-expire or archive them after N days.
  • Add a bulk archive/delete for drafts, and exclude abandoned drafts from the Dashboard count (or label it "unfinished drafts").

⚡Performance & efficiency

Wasted fetches, full-table scans, N+1 query fans, and unvirtualized DOM — all confirmed against real request volumes.

P1 Push notifier re-scans the whole leaves & books tables every 60 s

HighCodeperformance
WhatThe scheduler tick runs, per active user, _leaves_needing_action (pages all leaves 500 at a time, evaluates needs_action in Python) and submitter_name (a db.get(User) per book). A "compute once per tick" helper exists but the notify path doesn't use it.
ImpactOrg-wide full scans plus per-book point queries every minute, multiplied by active users — the single biggest steady-state load.
Plan
  • Compute the org-wide "needs action" leave set once per tick (the precompute path already exists — route actionable_items/_notify_user through it).
  • Express needs_action as a SQL WHERE so the scan disappears.
  • Batch-resolve submitter names with one IN query (see P4).

Method, scope & what was verified not to be a problem

How this was produced. The app was driven live at https://gssg.lan with Playwright, signed in as an administrator, across 14 routes (Dashboard, Employees + detail, Services/Application + form, Records/Books, Leaves report + mobile drawer, Ledger, Settings, Expiry, Duty Locations, Intake, Scan Inbox, Access Requests), plus mobile viewport (nav drawer, bottom tabs, leave drawer) and the Arabic/RTL toggle (verified dir=rtl, lang=ar, translated titles). Findings were cross-checked directly against data/gssg.db, and a full static review of the frontend (~64k LOC, React 19 + Vite + TanStack Query) and backend (FastAPI + SQLAlchemy) was run in parallel.

Safety. No destructive mutations were performed on the live data. A temporary admin account was created to log in through the real UI and was deleted afterward (the 7 real users are intact); a temporary audit session was minted and revoked.

Explicitly checked and clean: no bare except: (all except Exception with logging); no mutable default args; models are well-indexed for common filters; every v1 endpoint maps to a real frontend client call (no dead endpoints); Arabic/RTL rendering works; login lockout/attempts logic present. The main risk areas are the data layer (D1–D4) and steady-state notifier load (P1), not security or crashes.

🎯Suggested order of attack

4
